## Task 6 — Repo-level docs (9 files remaining)

Each file: path, role, content sketch, and any open decision.

### 6.1 `CODE_OF_CONDUCT.md`
- **Role:** Standardize community expectations. Linked from `CONTRIBUTING.md`.
- **Content:** Contributor Covenant 2.1, verbatim.
- **Open decision:** Contact email. Placeholder `community@sent.example` with a
  `<!-- REPLACE -->` marker unless overridden.

### 6.2 `docs/getting-started.md`
- **Role:** First page a new user hits after the README. Walks through a single
  end-to-end interaction.
- **Sections:**
  1. Prereqs (Claude Code installed)
  2. Two install paths (marketplace + local `file://`)
  3. Verifying install via `/plugin list`
  4. First task: `/waba-template` → author a shipping notification
  5. What skill activation looks like in Claude Code's UI
  6. Where to go next (per-skill SKILL.md links)

### 6.3 `docs/skill-anatomy.md`
- **Role:** The contract between contributors and the validator. Cited from
  `CONTRIBUTING.md`.
- **Sections:**
  - File location (`skills/<name>/SKILL.md`)
  - Frontmatter rules (name, description ≤1024 chars, "Use when" required)
  - Recommended section anatomy (Overview, When to Use, Process, Common
    Rationalizations, Red Flags, Verification) — equivalent headings allowed
  - Writing principles (process over knowledge, specific over general,
    anti-rationalization)
  - Cross-skill referencing via skill name not path
  - When to spill into `references/`

### 6.4 `docs/claude-code-setup.md`
- **Role:** Claude Code-specific install detail that doesn't fit in the README.
- **Sections:** Marketplace install, local plugin install, troubleshooting
  (SSH errors, plugin not appearing, skill not activating), checking context
  with `/context`, uninstall.

### 6.5 `docs/claude-ai-setup.md`
- **Role:** Equivalent for claude.ai users who can't use the Claude Code plugin.
- **Sections:** Zip individual skill → upload via Settings → Features → Skills,
  plan tier requirements, no-network caveat, multi-skill upload workflow.

### 6.6 `.github/ISSUE_TEMPLATE/bug_report.md`
- **Role:** Force structured bug reports.
- **Fields:** Affected skill, what you asked, what the agent answered, what
  you expected, Claude Code version, plugin version.

### 6.7 `.github/ISSUE_TEMPLATE/new-skill-proposal.md`
- **Role:** Catch ideas before they become half-built PRs.
- **Fields:** Proposed name, one-line description (the actual `description`
  field), trigger phrases, what existing skill it complements, why it's
  WABA-domain not generic engineering, sketch of the workflow.

### 6.8 `.github/ISSUE_TEMPLATE/skill-improvement.md`
- **Role:** Reports of out-of-date guidance (Meta changed their docs, a
  rejection code changed meaning).
- **Fields:** Affected skill, current text that's wrong, source of correction,
  suggested fix.

### 6.9 `.github/PULL_REQUEST_TEMPLATE.md`
- **Role:** Enforce structure for PRs.
- **Fields:**
  - Summary
  - Type (new skill / improvement / docs / tooling)
  - Validator output paste
  - Sample interaction transcript (most useful field for a skills repo)
  - Checklist (validator ran, description has "Use when", body ≤500 lines,
    no Meta-doc duplication, references updated)

  addyosmani has this; none of the 5 skills currently need shared personas.

  Worth it once we have ≥10 skills.

  using all 5 skills together. Better as a follow-up release.

  calls aren't painful enough yet to automate.
