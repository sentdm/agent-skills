## Authentication (v3)

Header-only auth, single key:

```http
x-api-key: <UUID>
```

No `x-sender-id` in v3 — that's v2 legacy. API key alone identifies the account.

Base URL: `https://api.sent.dm`

## Response envelope (all endpoints)

```json
{
  "success": true | false,
  "data": <T> | null,
  "error": { "code": "STRING", "message": "STRING", "details": {} | null, "doc_url": "STRING" | null } | null,
  "meta": { "request_id": "STRING", "timestamp": "ISO8601", "version": "v3", "response_time_ms": int? }
}
```

## Common response headers

- `X-Request-Id` — unique request ID; include in support tickets
- `X-Response-Time` — server processing time
- `X-API-Version` — always `v3`
- `X-RateLimit-Limit` — max requests per window
- `X-RateLimit-Remaining` — remaining in current window
- `X-RateLimit-Reset` — Unix timestamp of window reset
- `Retry-After` — seconds until retry (only on 429)
- `X-Sandbox: true` — present when request was sandbox mode
- `Idempotent-Replayed: true` — present when this response was replayed from idempotency cache
- `X-Original-Request-Id` — present on idempotent replays, references the original request

## Sandbox mode

Add `"sandbox": true` to any mutation request body. The API validates and returns a realistic fake response without side effects (no provider call, no DB write, no charge). Response includes `X-Sandbox: true` header.

Supported on all mutation endpoints: POST/PUT/PATCH/DELETE on `/v3/messages`, `/v3/contacts`, `/v3/templates`, `/v3/webhooks`, `/v3/profiles`, `/v3/brands`.

## Idempotency

Header: `Idempotency-Key: <key>` where key matches `^[a-zA-Z0-9_-]{1,255}$`. Cached for 24 hours, scoped per customer account.

Concurrent requests with the same key → second returns `409 CONFLICT_001`.

Replays: response includes `Idempotent-Replayed: true` and `X-Original-Request-Id: <original>`.

Same key + different payload → returns the cached response for the *first* payload, not a new one. Always use unique keys per distinct operation.

Supported endpoints (full list):
- POST `/v3/messages`
- POST, PATCH `/v3/contacts` and `/v3/contacts/{id}`
- POST, PUT, DELETE `/v3/templates`, `/v3/templates/{id}`
- POST, PATCH, DELETE `/v3/profiles`, `/v3/profiles/{id}`
- POST `/v3/profiles/{id}/complete`
- POST, PUT, DELETE `/v3/brands`, `/v3/brands/{id}`
- POST, PUT, DELETE `/v3/brands/{id}/campaigns`, `/v3/brands/{id}/campaigns/{id}`
- POST, PUT, DELETE, PATCH `/v3/webhooks`, `/v3/webhooks/{id}`
- POST `/v3/webhooks/{id}/rotate-secret`
- POST, PATCH, DELETE `/v3/users`, `/v3/users/{id}`

## Rate limits

**Standard endpoints** — 200 req/min, burst 50. Most endpoints.

**Sensitive endpoints** — 10 req/min, burst 5:
- POST `/v3/webhooks/{id}/rotate-secret`
- POST `/v3/users`
- POST `/v3/profiles/{id}/complete`

**Message sending** — tiered:
- Starter: 60 messages/min
- Growth: 300 messages/min
- Enterprise: custom

Webhook test: 60/min. All other GETs: 200/min.

Per-endpoint detail in the source doc's "Rate Limiting by Endpoint" section.

## Error envelope and full error code catalog

### Authentication

| Code | HTTP | Meaning |
|---|---|---|
| `AUTH_001` | 401 | Missing `x-api-key` header |
| `AUTH_002` | 401 | Invalid / unrecognized API key |
| `AUTH_004` | 403 | User role lacks permission for this operation |
| `AUTH_005` | 403 | Account valid + setup complete but pending final activation by Sent |
| `AUTH_006` | 403 | KYC not complete. Account states: `SIGNED_UP`, `KYC_STARTED`, `WHITELISTED`, `ONBOARDING_STARTED`, `KYC_RESUBMISSION_REQUESTED` |
| `AUTH_007` | 403 | KYC done but no messaging channel configured. Account states: `KYC_COMPLETED`, `MESSAGE_COMPLIANCE_COMPLETED` |

### Validation

| Code | HTTP | Meaning |
|---|---|---|
| `VALIDATION_001` | 400 | Generic request validation failed. `error.details` is `{field: [messages]}` |
| `VALIDATION_002` | 400 | Phone number not in E.164 (`+CCNNNNNNNNNN`) |
| `VALIDATION_003` | 400 | UUID malformed |
| `VALIDATION_004` | 400 | Required field missing |
| `VALIDATION_005` | 400 | Value out of range (e.g., `retry_count` must be 1-5; `timeout_seconds` must be 5-120) |

| Code | HTTP | Meaning |
|---|---|---|
| `CONFLICT_001` | 409 | Concurrent idempotent request in flight; retry after short delay |

### Service / Internal

| Code | HTTP | Meaning |
|---|---|---|
| `SERVICE_001` | 503 | Cache service unavailable |
| `INTERNAL_001` | 500 | Unexpected internal server error |
| `INTERNAL_002` | 500 | Database operation failed |
| `INTERNAL_003` | 502 | External SMS/WhatsApp/RCS provider error |
| `INTERNAL_004` | 504 | Internal timeout |

### Send-time error codes (per-message, not in HTTP response)

These appear in the `description` field of:
- `GET /v3/messages/{id}` (when `status=FAILED`)
- `GET /v3/messages/{id}/activities` (on the FAILED activity)
- The `message.failed` webhook payload (you must fetch the message to see the code)

| Code | Meaning |
|---|---|
| `ERR_CONSENT_BLOCKED` | Per-message consent block: contact has `opt_out=true` or is on phone-channel suppression list. No provider call, no charge. |
| `ERR_ROUTE_DENIED` | No active route could deliver to the requested channel/country. |
| `ERR_TEMPLATE_PARAMS_INVALID` | Required template variables missing or failed regex validation. |

**Important distinction:** `BUSINESS_004` (all-recipients opt-out) is synchronous and returns 422 on the batch. `ERR_CONSENT_BLOCKED` is per-recipient and surfaces async on individual messages while the rest of the batch succeeds.

## Message lifecycle

```
QUEUED → ROUTED → SENT → DELIVERED → READ (WhatsApp & RCS only)
  ↓        ↓       ↓        ↓
FAILED  FAILED  FAILED   FAILED
```

| Status | Meaning |
|---|---|
| `QUEUED` | Accepted, waiting to dispatch |
| `ROUTED` | Assigned to a carrier/provider |
| `SENT` | Dispatched to carrier/WhatsApp/RCS |
| `DELIVERED` | Confirmed delivery to device |
| `READ` | Recipient opened. WhatsApp + RCS only (SMS has no equivalent) |
| `FAILED` | Delivery failed; see `description` for code |
| `RECEIVED` | Inbound message from end user |

## Webhook payload format

```json
{
  "field": "message",
  "sub_type": "message.delivered",
  "timestamp": "2026-01-15T10:35:00+00:00",
  "payload": {
    "account_id": "<UUID>",
    "message_id": "<UUID>",
    "message_status": "DELIVERED",
    "channel": "sms",
    "inbound_number": "+1...",
    "outbound_number": "+1...",
    "template_id": "<UUID>"
  }
}
```

Top-level shape: `field`, `sub_type`, `timestamp`, `payload`. Sub-types follow `<field>.<event>` convention (e.g., `message.delivered`, `message.failed`, `message.read`).

## Channel selection (POST /v3/messages)

```json
{ "channel": ["sms", "whatsapp", "rcs"] }
```

- `["sms", "whatsapp", "rcs"]` — creates one message per channel; all dispatch
- `["sms"]` — SMS only
- omitted — automatic optimal selection

## Onboarding state machine (account-level)

In order:
1. `SIGNED_UP` (email verified)
2. `KYC_STARTED`
3. `WHITELISTED` (or `KYC_RESUBMISSION_REQUESTED` if KYC needs more docs)
4. `ONBOARDING_STARTED`
5. `KYC_COMPLETED`
6. `MESSAGE_COMPLIANCE_COMPLETED` (channel + brand/campaign work done)
7. (Activated) — full API access

API auth returns:
- 1-3: `AUTH_006` (KYC not complete)
- 5-6: `AUTH_007` (channel setup not complete)
- 6→7: `AUTH_005` (waiting for activation)

## RCS specifics (Sent-confirmed)

- Not self-service. Requires one-time carrier approval initiated by contacting Sent (`support@sent.dm`).
- Once approved, RCS Agent (branded sender) appears in dashboard.
- Falls back to SMS automatically for non-RCS-capable recipients.
- Use `"channel": ["rcs", "sms"]` to make fallback explicit.
- Glossary terms: RCS Agent, Rich Card, Carousel Card (up to 10 rich cards), Suggestion Chip (quick reply / open URL / dial number), Branded Sender.

## Webhook event lifecycle (verified from quickstart)

Webhook delivery for a message goes:
1. `message.queued`
2. `message.routed`
3. `message.sent`
4. `message.delivered`
5. `message.read` (WhatsApp / RCS only)

On failure at any stage: `message.failed` with `payload.message_status = FAILED` and the reason in the message detail (fetch via `GET /v3/messages/{id}`).

## Sandbox vs idempotency (both can stack)

Both can be combined: `sandbox: true` in body + `Idempotency-Key` header → validates request, returns fake response, caches it for 24 hours.

Use for: integration tests, CI/CD, debugging production payloads safely.
